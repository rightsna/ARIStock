# 🤖 AI Agent Guide: Flutter Coding Standards

이 문서는 AI 에이전트가 이 프로젝트 환경에서 코드를 작성할 때 반드시 준수해야 하는 **기술적 규칙 및 개발 표준**입니다.

---

## 1. 개발 환경 설정 (Environment Setup)
작업을 시작하기 전, 에이전트는 다음 환경이 준비되어 있는지 확인하고 미비할 경우 설치를 권장하거나 지원해야 합니다.

1.  **Flutter SDK**: 시스템에 Flutter가 설치되어 있어야 합니다. (`flutter doctor`로 확인)
2.  **Dart SDK**: Flutter 설치 시 함께 포함되나, 단독 실행이 필요한 경우 별도 설치 여부를 확인합니다.
3.  **Git 및 계정 연동**: Git이 설치되어 있어야 하며, GitHub 등 원격 저장소와 계정이 연동되어 있어야 합니다. **계정 연동이 확인되지 않을 경우 작업을 즉시 중단하고 사용자에게 연동을 요청하십시오.**
4.  **환경 설치 지침**:
    - Flutter/Git이 없는 경우: 공식 가이드를 안내하거나 시스템 패키지 매니저를 통해 설치를 지원합니다.
    - 필수 확장 프로그램: VS Code 사용 시 `Flutter`, `Dart` 확장이 설치되어 있는지 확인합니다.
5.  **Git 관리**: `build/`, `.dart_tool/` 및 민감 정보(`.env`)가 포함되지 않도록 `.gitignore`를 설정합니다.
6.  **배포 타켓 확인**: 개발 시작 전, 반드시 사용자에게 **어떤 플랫폼(Desktop, Web, Mobile)으로 배포할 것인지 질문하고 확인**을 받아야 합니다. 확정 전까지는 구현을 시작하지 마십시오.


---

## 2. 개발 철학 (Core Philosophy)
1.  **단순화 (Simplicity)**: 비개발자가 읽어도 직관적으로 이해할 수 있는 코드를 작성합니다. 복잡한 패턴보다 명확한 명명이 우선입니다.
2.  **격리 (Isolation)**: 각 기능(Component)은 완전히 독립된 폴더 내에서 완결되어야 합니다. 한 부분의 수정이 전체에 영향을 미치지 않도록 격리합니다.
3.  **비개발자 중심 (Citizen-friendly)**: 주석과 변수명은 사용자의 비즈니스 언어를 사용하며, 기술적 장벽을 최소화합니다.

---

## 3. 폴더 구조 규격 (Standard Directory Structure)
모든 기능 단위는 `components/` 하위에 독립된 폴더로 구성하며, 각 폴더는 다음 구조를 따릅니다.

```text
lib/
├── components/              # 독립된 기능 모듈 (격리 원칙)
│   ├── [feature_name]/      # 각 기능의 이름 (예: user, auth, search 등)
│   │   ├── screens/         # 해당 기능의 UI 화면들
│   │   ├── repositories/    # 데이터 소스 접근 (API, DB)
│   │   ├── providers/       # 상태 관리 로직 (ChangeNotifier)
│   │   ├── models/          # 데이터 규격 정의
│   │   └── services/        # 비즈니스 로직 및 통합 처리
│   └── ...
├── shared/                  # 전역 공통 자원
│   ├── theme.dart           # 앱 디자인 테마
│   ├── widgets.dart         # 공용 컴포넌트
│   └── utils.dart           # 공통 유틸리티
└── main.dart                # 앱의 진입점 및 환경 설정
```

---

## 4. 실무 지침 (Coding Instructions)
1.  **독립성 보장**: 특정 컴포넌트 폴더 하나만 복사해서 이동해도 작동할 수 있도록 외부 참조를 최소화하십시오.
2.  **명시적 상태 업데이트**: `provider` 패키지를 사용하며, 상태 변경 시 `notifyListeners()`를 명확히 호출하십시오.
3.  **친절한 주석**: 파일 상단과 복잡한 로직 위에는 비개발자가 이해할 수 있는 수준의 한글 주석을 필수적으로 작성하십시오.
4.  **사용자 친화적 에러 처리**: 모든 예외 상황은 비개발자가 당황하지 않도록 화면에 쉬운 말로 안내하십시오.

---

## 5. 배포 가이드라인 (Deployment)
개발 완료 후 배포 시 다음의 기준을 따르거나 사용자에게 선택지를 제공하십시오.

1.  **지원 플랫폼**: 데스크탑(macOS, Windows), 웹, 모바일(iOS, Android).
2.  **버전 관리 (선택)**: 모든 주요 기능 구현 완료 후 사용자가 원할 경우 **Git Repository**에 업로드하여 히스토리를 관리하십시오.
3.  **웹 배포 (선택)**: 웹 타겟의 경우, 빠르고 간편한 확인을 위해 **Vercel**을 통한 자동 배포 환경 구축을 지원하십시오.
